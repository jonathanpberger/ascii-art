# ADR 0002: ASCII Compliance Testing

- **Status:** Accepted
- **Date:** 2026-05-06
- **Authors:** Tutorial Author

## Context

`0001.ADR.ASCII.md` pinned down the shape, size, and shell of every ASCII art
markdown file in `ascii/`. Those rules are deliberately narrow so that they
are *mechanically* checkable — count the headings, measure the code block,
inspect the filename. No judgment required.

That property is wasted if the only enforcement happens at human or AI
review time. Reviewers — even fast vendor-diverse reviewer agents — are an
expensive late gate. A pull request that violates a rule from
`0001.ADR.ASCII.md` should fail before any reviewer is paged: each round-trip
through a reviewer costs wall-clock time, model spend, and the submitter's
attention. In the worst case, a polecat opens a PR with eleven trivial
violations and a reviewer agent burns a full pass writing them up.

Continuous integration is the right shape for this gate. CI is cheap, fast,
runs on every PR, and reports a single binary signal. Branch protection on
`main` can require that signal to be green before merge. Reviewer agents
(page 04) and human reviewers then spend their attention on substantive
judgment — taste, rhyme quality, alphabet coverage — instead of counting
columns.

## Decision

Every pull request is gated by a GitHub Actions workflow that enforces the
rules locked in `0001.ADR.ASCII.md`.

### Workflow location and trigger

1. The workflow lives at `.github/workflows/ascii-compliance.yml`.
2. It triggers on `pull_request` events whose changed paths match
   `ascii/**`.
3. Its job ID — and therefore its required-status-check name — is
   `ci/ascii-compliance`.

### Required checks

The workflow MUST run all six checks below. Any single failure fails the
workflow.

1. **Filename regex.** Every changed file under `ascii/` matches
   `^(?:[a-z]|[1-9][0-9]?|100)\.md$`. Letters are single lowercase
   `a`–`z`; numbers are decimal `1`–`100` with no leading zeros.
2. **Location.** Each changed file sits directly in `ascii/` at the
   repository root. No subdirectories.
3. **Single H1 with character title.** The file has exactly one H1.
   For letters its text matches `^# [A-Z]$`; for numbers it matches
   `^# [1-9][0-9]?$|^# 100$`.
4. **Single fenced `text` code block.** The file has exactly one fenced
   code block, opened with ```` ```text ```` and closed with ```` ``` ````.
5. **ASCII art bounds.** Inside that code block: at most 8 lines of
   content (excluding fences), at most 20 columns wide (longest line),
   every character a printable ASCII code point in the range `0x20`–`0x7E`,
   no tab characters, and no trailing whitespace on any line.
6. **Two-line rhyme couplet.** Below the closing fence the file has
   exactly two non-empty lines and nothing else — no extra headings,
   blockquotes, lists, links, HTML, or trailing content.

### Lockstep with branch protection

The required-status-check name `ci/ascii-compliance` is the contract between
this workflow and branch protection. Page 03 of the tutorial originally ran
`branch-protection.sh` with `STATUS_CHECKS=""` — protection was on, but no
specific check was wired in. Hardening 1 instructs learners to re-run
`branch-protection.sh` with `STATUS_CHECKS="ci/ascii-compliance"` once this
workflow is committed. From that moment forward, `main` will refuse merges
whose `ci/ascii-compliance` job has not reported success.

## Consequences

**Positive.** A cheap, deterministic mechanical gate runs before any
reviewer sees the PR. Bad PRs fail fast with a single clear signal. Human
and AI reviewers are freed up for substantive judgment — taste, rhyme,
alphabet coverage — rather than counting columns. Hardening 4's
vendor-diverse reviewers can treat the green CI check as a precondition
and focus on what models can disagree about.

**Negative.** The workflow is code, and code that mirrors a spec needs
maintenance. Every amendment to `0001.ADR.ASCII.md` — for example, a future
relaxation of the 8x20 canvas — requires a synchronized change to
`.github/workflows/ascii-compliance.yml`. The two artifacts MUST move
together; a drifted workflow either blocks valid PRs or rubber-stamps
invalid ones.

**Forward.** The reviewer agent introduced in page 04, and the
vendor-diverse reviewer rig added in Hardening 4, both consume CI's
pass/fail signal as a starting point. They cite ADR 0001's rule numbers in
their comments while CI enforces those same rules mechanically.

## Compliance check

This ADR is enforced by `.github/workflows/ascii-compliance.yml`, authored
as a separate deliverable in Hardening 1 alongside this document. The two
files are a matched pair: this ADR is the canonical statement of *what* CI
must check; the workflow is the executable implementation. The reviewer
agent on page 04 cites rule numbers from `0001.ADR.ASCII.md`; the CI
workflow enforces the same rules without judgment. A PR is merge-eligible
on `main` only when the `ci/ascii-compliance` status check is green.
